01Context file#

Bộ nhớ dài hạn của dự án

File agent tự đọc ở đầu mỗi session. Giúp agent hiểu context dự án mà không cần giải thích lại mỗi lần.

Trạng thái Kết quả
❌ Không có file Mất 5–10 phút mỗi session Giải thích lại tech stack, convention, quyết định thiết kế.
✅ Có file tốt Agent bắt tay vào việc ngay, đã nắm platform, rules, cấu trúc
🔄 Maintain Living document, update sau mỗi sprint / quyết định kiến trúc. Cần review định kỳ

Context file được nạp theo scope từ rộng đến hẹp — file càng gần nơi đang làm việc thì càng cụ thể: 

~/.claude/CLAUDE.md          → global: phong cách cá nhân, áp dụng mọi project
<repo>/CLAUDE.md             → project: tech stack, convention của repo
<subfolder>/CLAUDE.md        → rule riêng cho module/folder (chỉ khi thực sự cần)
Đừng phân tán quá sớm

Chỉ tách context file theo folder khi có nhu cầu rõ ràng. Quá nhiều file context rải rác làm dev khó biết rule nào đang áp dụng.

Nguyên tắc vàng
  • Chỉ dẫn cần luôn áp dụng ở mọi turn → đặt vào CLAUDE.md
  • Quy trình thỉnh thoảng mới cần → làm thành Skill
  • Script phải chạy tự động → làm thành Hook

Đừng nhồi mọi thứ vào CLAUDE.md.

Ví dụ template CLAUDE.md cho dự án

# [App Name]

## Project Overview

[One-liner: workflow (bare / Expo CNG / managed), target platforms, CI/CD, release method]

## Tech Stack

- Framework: [React Native version, Expo SDK if any]
- Language: [TypeScript config,  strict or not]
- State: [global state lib + server state lib]
- Navigation: [navigation lib]
- Styling: [styling approach]
- Key integrations: [API client, SDKs the project depends on]

## Conventions

- File naming: [component / hook / util convention]
- Commit: [format, e.g. type(scope): message]
- Branch: [naming pattern, e.g. feature/TICKET-###]
- Package install: [npm / yarn / pnpm,  và `npx expo install` cho Expo deps]

## Important Rules,  NEVER

- [Project-specific hard rules]

## Current Status

- Sprint / milestone: [...]
- Known issues: [...]
Tips

Nhờ agent gen file này (/init hoặc "đọc codebase rồi viết CLAUDE.md"), sau đó dev review + sửa.

02Skill#

Đóng gói quy trình, tự kích hoạt

Skill không phải tool runtime như MCP. Skill là package gồm instruction + metadata + optional files/scripts, kích hoạt on-demand. Giúp agent tập trung vào nhiệm vụ, không lan man, tải context thừa.

Về security

Skill có thể khai báo allowed-tools để giảm friction khi active, nhưng đây không phải security boundary. Chặn hành vi nguy hiểm cần dùng permission settings, hook hoặc policy ở cấp repo/team (xem mục 05 và 14).

Hoạt động theo 2 giai đoạn:

Giai đoạn Hành vi
Lúc khởi động Chỉ load name + description, agent đọc để biết skill dùng khi nào
Khi đúng việc Load full SKILL.md, agent đọc trọn quy trình rồi làm theo
Vì sao description quan trọng

Agent chủ yếu dựa vào name + description để quyết định skill nào phù hợp. Description phải nêu rõ: dùng khi nào, input là gì, output mong muốn là gì. Mô tả mơ hồ → skill không được gọi. Agent tự gọi khi mô tả khớp, hoặc dev gọi trực tiếp bằng /tên-skill.

Ví dụ description tốt:

Use this skill when a developer finishes a React Native feature and needs a
self-test checklist before handing it to QA. Covers happy path, edge cases,
negative cases, and regression risks.

Skill nên dùng

Bộ skill superpowers - Obra

Ép agent làm việc theo một trình tự chuẩn: dừng lại hỏi spec → brainstorm → lên plan → viết test trước (TDD) → code → tự review 2 vòng (đúng spec + chất lượng code).

Giải quyết failure mode #1 của AI: code vội, nhìn thì ổn nhưng không ổn định, nhiều bug ẩn khi run thật.

react-native-best-practices - Callstack

Tối ưu React Native: chặn JS thread, Hermes, giảm TTI, FlashList, phát hiện memory leak. Agent tự tham chiếu pattern này khi review/optimize codebase React Native.

expo - Expo Official Team

Cho repo Expo: build native UI, viết module, deploy, upgrade... Cung cấp cho agent conventions của Expo, tăng chất lượng code và độ chính xác khi thực hiện các tác vụ liên quan Expo.

skill-creator- Anthropic Official

Dùng để tạo ra các skill khác. Thay vì tự học cú pháp SKILL.md, chỉ cần mô tả công việc bằng ngôn ngữ tự nhiên, nó hỏi lại để làm rõ, draft skill, test thử, rồi đóng gói thành skill chạy được.

Case study: bss-jira-task-summary

Vấn đề: Mỗi lần nhận Jira task, phải đọc cả description lẫn toàn bộ comment (comment chứa quyết định của khách + blocker kỹ thuật). Nhờ agent tóm tắt thì mỗi lần ra một format khác nhau, hay bỏ sót comment quan trọng.

Cách làm với skill-creator: Mô tả đúng nhu cầu, "khi tôi paste một Jira task, hãy đọc cả description và comment theo thứ tự thời gian, phân loại comment (PA note / quyết định khách / phân tích dev / blocker), rồi xuất ra 2 phần cố định: yêu cầu task và hướng implement kèm file path cụ thể". Skill-creator hỏi lại để làm rõ output format, rồi sinh ra skill hoàn chỉnh.

Kết quả: Giờ mỗi lần /bss-jira-task-summary + paste ticket là ra đúng một format, hiểu rõ requirement, cover tốt các case và tránh miss thông tin.

03Subagent#

Context riêng, clean main session

Subagent là một instance Claude riêng biệt, có context window độc lập, chỉ trả về kết luận cho session chính. Dùng khi một task phụ có thể làm ngập session chính với log, kết quả search, nội dung file mà bạn không cần xem lại.

Khi nào dùng Subagent

Use case Ví dụ
Research nhiều file "Đọc toàn bộ module lớn rồi tóm tắt cách nó hoạt động"
Review code độc lập Fresh review không bị bias bởi code nó vừa viết
Deep Inspect Check lib docs, code - so sánh approach mà không nhồi vào context chính
Quan hệ với context

Subagent là một best practice để giữ context chính dưới 40%. Việc nào "đọc nhiều, dùng ít" → đẩy sang subagent.

Quản lý context

Context window có giới hạn. Càng đầy, agent càng dễ "quên" yêu cầu ban đầu và code loạn dần, thiếu chính xác, gọi là context rot.

Ngưỡng Trạng thái
< 40% ✅ Tốt
60–70% ⚠️ Vùng nguy hiểm, cần xử lý

3 lệnh cần nhớ:

Lệnh Dùng khi Ghi chú
/compact Vẫn cùng một task và muốn tiếp tục Nén lịch sử, giữ phần quan trọng, bỏ phần thừa
/clear Chuyển sang task khác Xoá sạch context, bắt đầu mới
/rewind (hoặc Esc ×2) Agent đi sai hướng, đã sửa lung tung Quay về checkpoint trước khi agent break
Phân biệt nhanh

/compact = cùng task, tiếp tục · /clear = task mới, làm lại · /rewind = undo agent đi sai.

04MCP#

Cổng USB-C của Agent

Model Context Protocol là chuẩn mở (Anthropic) chuẩn hoá cách agent kết nối tới hệ thống thật bên ngoài. Gỡ thế cô lập: thay vì chỉ biết những gì trong prompt, agent thao tác được với simulator, docs latest, database thật...

  • Khai báo → server trong file config (.mcp.json)
  • Lúc chạy → agent tự gọi tool khi cần; có thể manual gọi thủ công, tương tự Skill

MCP nên sử dụng

Argent - Software Mansion

Điều khiển iOS Simulator + Android Emulator: chạy, tap, profile, đọc log. Support test, debug và bổ sung context cho agent trong quá trình phát triển feature và fix bug.

Context7

Tra docs thư viện/SDK đúng version (Expo SDK, React Native library, 3rd-party SDK). Giải quyết vấn đề agent lấy dữ liệu sai do mismatch version, dẫn đến đi sai hướng và phát sinh lỗi không đáng có.

Team ưu tiên MCP mode để agent gọi docs trực tiếp trong workflow.

Notes về Context7
Cách prompt Context7 cho chuẩn

Khi task phụ thuộc version, nói rõ library và version:

use context7, Expo SDK 53
use context7, React Navigation v7
use library /expo/expo for EAS Update docs

Càng rõ library/version, agent càng ít lấy nhầm API cũ.

Context7 không thay thế review cuối

Context7 giúp agent lấy docs mới hơn, giảm hallucination, nhưng không phải nguồn chân lý tuyệt đối. Với phần critical như auth, payment, release, native config hoặc migration lớn — dev vẫn cần verify lại bằng official docs/source repo trước khi merge.

Context7 setup

Recommend npx ctx7 setup (interactive, chọn CLI + Skills hoặc MCP mode). API key là khuyến nghị để có rate limit tốt hơn. Nếu setup manual MCP, dùng server URL chính thức và cấu hình API key ở user scope / secret manager, không commit vào repo.

05Hook#

Script deterministic tại lifecycle

Hook là script chạy tự động tại các thời điểm trong vòng đời session. Khác Skill/MCP ở chỗ: hook không phải AI, nó là logic code chạy, không phụ thuộc agent "nhớ" hay không.

Các event chính

Event Thời điểm Use case
PreToolUse TRƯỚC khi agent gọi tool Chốt chặn an toàn quan trọng nhất
PostToolUse SAU khi tool xong VD: TypeScript check sau mỗi edit
Stop Khi session kết thúc VD: Bắt buộc test pass trước khi "done"

Exit codes

Code Ý nghĩa
0 Hook chạy ổn, không chặn
2 Blocking error, chỉ chặn với event có thể chặn như PreToolUse. Với PostToolUse (tool đã chạy xong) thì chỉ báo lỗi, không chặn ngược được
non-zero khác Lỗi không chặn, log/hiển thị tuỳ event
Lưu ý

Đừng hiểu "exit 2 = chặn hoàn toàn ở mọi nơi". Muốn ngăn một việc xảy ra, hook phải đứng trước việc đó (PreToolUse).

Hook quan trọng cho team, secret-scan

Khác Skill/MCP, hook không cài từ marketplace, dev sẽ cần tự viết. Với mobile dev, nên có hook chặn commit có chứa secret (key, token, .env). Đây là chốt chặn deterministic cho rủi ro lộ key — thứ mà chỉ nhắc trong CLAUDE.md không đảm bảo được.

Setup 3 bước

Bước 1, Viết script scan: .claude/hooks/secret-scan.sh 

#!/usr/bin/env bash
set -euo pipefail

# Hook receives JSON via stdin from Claude Code.
# Script inspects the command to only scan before `git commit`.
input="$(cat)"
cmd="$(printf '%s' "$input" | jq -r '.tool_input.command // ""')"

# Skip other bash commands to avoid slowing down the workflow.
if [[ "$cmd" != git\ commit* ]]; then
  exit 0
fi

# Only scan added lines to avoid false positives when removing legacy secrets.
diff="$(git diff --cached --unified=0 | grep -E '^\+' | grep -vE '^\+\+\+' || true)"

# Common patterns,  extend per the SDKs your team uses
patterns=(
  'shpat_[A-Za-z0-9]{32,}'                                   # Shopify token
  'AIza[0-9A-Za-z_-]{35}'                                    # Firebase / Google API key
  '-----BEGIN[A-Z ]*PRIVATE KEY-----'                        # private key (.p8/.pem)
  '([Ss]ecret|[Tt]oken|[Pp]assword)[_-]?[Kk]ey?["'\'' :=]+["'\''][^"'\'' ]{8,}'  # generic
)

for p in "${patterns[@]}"; do
  if echo "$diff" | grep -nE "$p" >/dev/null; then
    echo "Secret detected in staged changes. Commit blocked." >&2
    echo "Pattern: $p" >&2
    echo "Move the secret out of the repo (.env + .gitignore) and try again." >&2
    exit 2   # exit 2 = blocking error at PreToolUse → blocks the commit
  fi
done

exit 0
chmod +x .claude/hooks/secret-scan.sh

Bước 2, Đăng ký hook trong .claude/settings.json: 

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/secret-scan.sh"
          }
        ]
      }
    ]
  }
}

Bước 3: Test thử git add một file có chuỗi giống token rồi nhờ agent commit, hook phải chặn với exit 2.

Notes về secret-scan hook
Hook thay cho lời nhắc trong CLAUDE.md

Đừng viết "NEVER commit .env" trong CLAUDE.md rồi hy vọng agent nhớ. Hook scan chạy chắc chắn mọi lần, cho mọi dev, không phụ thuộc trí nhớ của agent.

Hook không phải lớp bảo vệ duy nhất

secret-scan chặn commit. Nhưng secret thật vẫn phải để ngoài repo (.env + .gitignore) và deny Read trong settings (xem mục 14). Hook là một lớp, không phải tất cả.

06Plugin#

All-in-one

Plugin là gói đóng sẵn chứa nhiều Skill + Subagent + Hook + MCP + slash command.

Ví dụ: Superpowers là một plugin, bên trong gồm nhiều skill (brainstorm, plan, TDD, review) + hook session-start, không phải một skill đơn lẻ.

Plugin nên dùng

superpowers - Obra

Đã nêu ở mục Skill. Nhắc lại để làm rõ: nó là plugin, không phải skill đơn lẻ. Bên trong gồm nhiều skill (brainstorm, plan, TDD, review) + hook session-start tự kích hoạt.

callstack agent-skills - Callstack

Marketplace của Callstack, chứa react-native-best-practices và các skill React Native khác. Cài cả marketplace rồi chọn skill cần.

Plugin nội bộ team (định hướng)

Khi bss-jira-task-summary, hook secret-scan, và các skill React Native team tự thiết kế đã ổn định, có thể đóng gói thành một plugin nội bộ để tối ưu workflow.

Security policy khi dùng MCP / Hook / Plugin

MCP, hook và plugin là executable code, không phải docs/config thường.

Important

Áp dụng các quy tắc bắt buộc sau:

  • Không commit API key, token, credential vào .mcp.json, dùng .mcp.json.example nếu cần share
  • Không bật auto-approve cho toàn bộ MCP server của project
  • Không cài MCP/plugin từ nguồn lạ - chưa được kiểm duyệt; pin version khi có thể
  • Mọi thay đổi .mcp.json, .claude/settings.json, hook hoặc plugin nội bộ phải qua dev lead dự án review

10Nhận & đọc task#

Mục tiêu: Hiểu đúng requirement, liệt kê câu hỏi clarify, nhận diện dependency & risk trước khi code.

Các bước

  1. Đưa task cho agent phân tích, paste mô tả Jira + comments (hoặc screenshot)
  2. Nhận output có cấu trúc, yêu cầu task (what) + tình trạng (đã làm / blocked / chưa làm) + hướng implement (how)
  3. Chốt câu hỏi clarify, gom các điểm mơ hồ thành list gửi BA/PA trước khi bắt đầu
  4. Estimate, dùng baseline của agent, tự cộng buffer cho phần agent chưa nắm (tech debt, tích hợp bên thứ ba, performance)

Công cụ hỗ trợ

Tool Bước Vai trò
Skill bss-jira-task-summary 1–2 Output luôn đúng format, không bỏ sót comment. Comment chứa quyết định của khách + blocker kỹ thuật, bỏ qua = làm sai
Skill superpowers (brainstorming) 3 Với task lớn/mơ hồ, tự hỏi lại spec thay vì đoán, giúp lộ ra câu hỏi clarify mình chưa nghĩ tới
Rule of thumb khi estimate

Agent cho baseline, bạn chốt. Đừng lấy nguyên con số agent đưa, nó không biết tech debt và lịch sử dự án.

Kết quả
✕ Đọc một mình Bỏ sót edge case, không biết hỏi gì, estimate thiếu buffer
✓ Đọc cùng agent Highlight ambiguity ngay, có sẵn list clarify, nắm risk trước khi code

11Làm task (plan → code)#

Mục tiêu: Đảm bảo implement đúng hướng ngay từ đầu, không đi sai context dẫn đến lãng phí tài nguyên và đưa ra kết quả kém chất lượng.

Core: Plan trước → review plan → mới code → code từng phần nhỏ.

Các bước

  1. Tạo branch & gom context — git checkout -b /APP-###; tập hợp file/context liên quan
  2. Agent lên plan trước, đưa yêu cầu + context → agent đề xuất plan → dev review kỹ → confirm → mới code. Task lớn: ghi plan ra plan.md để giữ qua session
  3. Implement từng phần nhỏ, theo checklist trong plan. Mỗi phần xong → review & confirm → sang phần tiếp
  4. Review code agent vừa viết, đọc kỹ, hoặc giao subagent review độc lập
  5. Commit & cập nhật docs, message rõ; sửa CLAUDE.md nếu đổi convention/kiến trúc

Công cụ hỗ trợ

Tool Bước Vai trò
Plan mode (Shift+Tab hoặc /plan) 2 Agent explore & đề xuất, không sửa code cho tới khi bạn duyệt.
MCP context7 3 Khi code đụng Expo/React Native lib/3rd SDK, thêm use context7 để agent tra docs đúng version, tránh viết API deprecated
Skill superpowers 2–4 Ép quy trình plan → TDD → review, hạn chế lỗi agent thường mắc
Skill react-native-best-practices 4 Review góc độ performance: re-render, FlashList, memory leak
Subagent 4 Fresh review code, không bị bias bởi code nó vừa viết. Giữ context chính sạch
Plan mode vs Superpowers, khác gì?
  • Plan mode = human-driven: bạn chủ động bắt agent dừng để duyệt plan
  • Superpowers = agent-driven: agent tự enforce kỷ luật, không để skip bước kể cả khi bạn quên

Hai cái bổ trợ, không thay thế nhau.

12Fix bug#

Mục tiêu: Hiểu gốc rễ, fix một lần không tái phát. Đừng chỉ dán error message, cung cấp đủ context ngay từ đầu.

Công thức debug 5 thành phần

# Thành phần Nội dung
1 Error Message đầy đủ + stack trace
2 Reproduce Các bước tái hiện
3 Expected vs Actual Mong đợi vs thực tế
4 Recent changes File/feature vừa sửa
5 Environment Platform, React Native/Expo version, device, build variant

Các bước

  1. Reproduce trước khi fix, chưa tái hiện được thì chưa fix. Nhờ agent tạo test case tái hiện
  2. Phân tích root cause, dùng công thức 5 thành phần, tìm nguyên nhân gốc không chỉ symptom
  3. Xem 2–3 fix options, so sánh trade-off (safety / effort / risk) trước khi chọn
  4. Implement & review fix, nhờ agent review để chắc không gây side effect
  5. Test regression, verify fix không break feature khác

Công cụ hỗ trợ

Tool Bước Vai trò
MCP Argent 1, 5 Agent tự chạy app trên simulator để tái hiện bug và verify fix. Tap/swipe theo bước reproduce, đọc log runtime
Skill superpowers (systematic-debug) 2 Quy trình debug có hệ thống, chống nhảy vào fix symptom
Subagent 2 Bug phức tạp cần đọc nhiều file: giao subagent inspect, trả về danh sách nghi vấn, giữ main context sạch
Nếu bug đến từ production

Khi team dùng crash reporting (Sentry, Firebase Crashlytics), nhà cung cấp thường có MCP để agent đọc thẳng crash report + stack trace. Verify cách cài theo official docs trước khi thêm.

⎋ Tips

Agent đi sai hướng → ngắt ngay (ESC). Thu hẹp scope, đừng tốn token theo hướng sai. Checkpoint sai → /rewind về trước khi agent làm hỏng.

13Self-test trước khi giao tester#

Mục tiêu: Catch bug ngay ở khâu self-test để không bỏ sót test case quan trọng trước khi chuyển tester, tiết kiệm thời gian cho cả tester lẫn dev.

4 nhóm test bắt buộc

Nhóm Mô tả
Happy path Luồng chính chạy đúng
Edge cases Giá trị biên, dữ liệu bất thường
Negative cases Input sai, app xử lý lỗi đúng cách
Regression Không break feature liên quan

Các bước

  1. Agent sinh checklist: phủ đủ 4 nhóm dựa trên feature vừa làm
  2. Chạy self-test: tự test theo checklist, hoặc để agent chạy trên simulator
  3. Review code lần cuối: chạy structured review trước khi commit
  4. Chuẩn bị handoff note cho tester: xem template dưới

Công cụ hỗ trợ

Tool Vai trò
MCP Argent Agent tự chạy qua các case trên simulator: tap theo happy path, nhập edge value, chụp screenshot kết quả
/review Structured code review tại chỗ: bug, edge case, design issue. Nhanh, chạy local
/ultra-review Trước khi merge thứ critical (auth, payment, migration)
Skill self-test nội bộ (Todo) Team có thể chuẩn hóa một bộ skill sinh checklist theo đúng 4 nhóm + đặc thù dự án

Template handoff note cho tester

## [APP-###] Tên feature

Branch: /APP-###

Đã test:
- Happy path: [tóm tắt case đã pass]
- Edge/Negative: [case đã cover]
- Platform: iOS ✓ / Android ✓

Known limitations / chưa cover:
- [thứ tester cần chú ý, hoặc "không có"]

Cần test kỹ:
- [vùng rủi ro nhất, vd: flow thanh toán, deep link]
Phép tính đơn giản
  • Bug bắt ở self-test = ~5 phút
  • Bug do tester tìm ra = thêm ~30 phút (re-open → fix → re-test → close)

Self-test kỹ là đầu tư lời nhất.

14Security#

Xuyên suốt mọi workflow

Mục tiêu: Để agent access codebase một cách an toàn.

Khác các mục trên, security không phải workflow tuần tự, nó là lớp enforce một lần bằng config, áp dụng cho mọi workflow phía trên.

Ba nguy cơ chính

Nguy cơ Mô tả
Prompt injection Nội dung lạ (issue, MCP output, comment trong dependency) giấu chỉ dẫn; agent đọc rồi làm theo như lệnh thật
Supply chain (slopsquatting) Agent hallucinate tên package không có thật; attacker đăng ký sẵn tên đó thành package độc
Secret leak Agent đọc .env/key/token rồi tuồn ra qua curl, git push, hoặc markdown link

Lethal trifecta, Nguyên nhân gốc (Simon Willison)

Agent thành nguy hiểm khi có đủ 3 thứ cùng lúc. Không phải do model, nó chỉ làm đúng việc là nghe theo chỉ dẫn: 

1. Private data      -  đọc được source, .env, key, database
2. Untrusted content -  tham chiếu nguồn không đáng tin từ bên ngoài: web, issue, MCP output
3. Exfiltration      -  có đường gửi ra ngoài: curl, git push, gọi API, render link
Warning

Đủ ba yếu tố = vấn đề an toàn nghiêm trọng. Cách xử lý: chặn ít nhất một yếu tố trong mọi execution path. Meta gọi là Rule of Two, đừng để một session ôm cả ba thứ.

Hướng bảo vệ, Enforce bằng config, không bằng lời nhắc

Lớp Làm gì Tool
Permission deny → ask → allow: khoá hành động nguy hiểm bằng denylist, hỏi với npm/npx .claude/settings.json permissions
Secret scan Chặn commit chứa key/token Hook secret-scan (PreToolUse)
Secret storage deny Read không chặn cat qua Bash → để secret ngoài repo .gitignore
Con người Code agent phải review; MCP/device chỉ trỏ staging không trỏ production Review thủ công

Chốt chặn tối thiểu trong .claude/settings.json

Ở root repo, tạo hoặc cập nhật .claude/settings.json để chặn agent đọc trực tiếp các file secret phổ biến: 

{
  "permissions": {
    "deny": [
      "Read(**/.env)",
      "Read(**/.env.*)",
      "Read(**/*.p8)",
      "Read(**/*.p12)"
    ]
  }
}
Warning

deny Read chưa đủ — Permission deny giúp chặn Read tool và một số file command Claude Code nhận diện, nhưng đây không phải OS-level sandbox. Một subprocess như script Python/Node vẫn có thể tự đọc file nếu Bash được phép và môi trường không bị sandbox.

Secret thật phải untrack khỏi repo .gitignore.

Mặc định không cho agent đụng production data

Nếu cần đọc dữ liệu thật để debug, phải có approval rõ ràng, chỉ dùng quyền read-only và scoped theo đúng issue. Đặc biệt quan trọng nếu team cắm MCP của Sentry, Firebase hay Braze.

Checklist trước khi commit code có AI tham gia

  • Đã review toàn bộ code agent viết, không blind merge
  • Không có key/token/secret trong diff (hook đã scan, nhưng vẫn cần dev review check)
  • Package agent thêm vào là package có thật (check npm, không tin mặc định tên package agent gợi ý)
  • MCP đang dùng trỏ staging/dev, không phải production
Nguyên tắc

Phá đứt một cạnh của trifecta ở mọi execution path, rồi enforce bằng config. Giữ agent up-to-date (nhiều CVE cho bypass permission).

15Anti-patterns#

Những cái cần tránh

Khi nào KHÔNG nên dựa vào AI

AI là công cụ tăng tốc, không thay thế phán đoán. Những việc dev phải tự chủ:

  • Quyết định kiến trúc lớn: chọn New Arch vs Old Arch, đổi state management, cấu trúc navigation. Agent tư vấn được, nhưng quyết định là của dev.
  • Logic security-critical: auth flow, xử lý token, mã hoá. Review cực kỹ, không blind merge
  • Review cuối trước production: đặc biệt code đụng payment, user data, migration
  • Đánh giá trade-off có tính business: agent không biết deadline, ưu tiên khách hàng, lịch sử dự án

Các lỗi thường gặp khi dùng AI

Lỗi Vì sao nguy hiểm Thay vào đó
Merge code không đọc Agent viết code nhìn ổn nhưng sai logic/edge case; lỗi lọt thẳng vào production Luôn review từng phần. Code đụng auth/payment review cực kỹ
Dán cả file 1000 dòng vào chat Tốn context, nhiễu, agent mất focus Để agent tự đọc file (chỉ trỏ tên / @mention). Đây là agentic tool, không phải chat web
Một session cho cả ngày Context rot, agent quên yêu cầu ban đầu, code loạn dần Context > 50% hoặc đổi task → /compact hoặc mở session mới. Giữ context < 40%
Tin tên package agent đưa Slopsquatting, agent bịa tên package, attacker đăng ký sẵn tên đó thành malware Verify package có thật trên npm trước khi install
Để agent tự quyết kiến trúc Agent không biết tech debt, lịch sử dự án, ràng buộc business Agent tư vấn, dev quyết. Đặc biệt: New Arch vs Old Arch, state management
Nguyên tắc

Mọi lỗi trên đều quy về một nguyên nhân: giao cho agent nhiều quyền hơn mức mình kiểm soát được. Dùng AI để tăng tốc độ, nhưng luôn phải giữ vai trò là người kiểm định, đánh giá chặt chẽ ở mọi bước trong mọi qui trình.
